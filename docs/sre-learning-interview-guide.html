1. Beginner to Advanced Path

Beginner

  • Understand latency, traffic, errors, and saturation.
  • Learn the difference between logs, metrics, and traces.
  • Read the golden signals and SLO dashboards.
  • Understand what an SLI, SLO, and error budget mean.

Intermediate

  • Run guided load and capacity exercises.
  • Decide whether the bottleneck is app-tier or DB-tier.
  • Interpret alert rules and budget burn.
  • Use retries, timeouts, circuit breaking, and fallbacks.

Advanced

  • Build incident narratives from telemetry.
  • Design meaningful SLIs and production-style SLOs.
  • Use observability to justify scaling, tuning, or mitigation decisions.
  • Explain tradeoffs between reliability, performance, and cost.

2. Core SRE Concepts

Golden Signals

  • Latency: How long requests take.
  • Traffic: How many requests the system handles.
  • Errors: How often requests fail.
  • Saturation: How close the system is to exhaustion.

SLI / SLO / Error Budget

  • SLI: A measured indicator such as success ratio or latency under a threshold.
  • SLO: A target such as 99 percent availability.
  • Error Budget: The amount of failure or slowness you can spend before breaching the SLO.

4. Alerting

The lab includes alert rules for:

  • App latency SLO breach
  • Fast error-budget burn
  • Database latency
  • Database connection pressure
  • Resilience activity such as fallbacks and circuit openings

5. Resilience Patterns

  • Timeouts: Stop waiting forever on a slow dependency.
  • Retries: Recover from transient failures carefully.
  • Circuit breaker: Stop hammering an unhealthy dependency.
  • Fallback: Return a degraded but useful response when full success is not possible.

6. Observability-Driven Decisions

Examples from this lab:

  • High app latency with low event loop saturation and high DB latency points to the database tier.
  • High app latency with high event loop saturation and healthy DB suggests app-tier scaling or tuning.
  • Rising error rate plus circuit-breaker activity suggests dependency instability.

7. Capacity Planning Logic

Good capacity planning asks:

  • What throughput can the system sustain while still meeting the SLO?
  • What saturates first?
  • When does scaling help?
  • When does tuning help more than scaling?

Typical decision rules in this lab

  • Scale app containers first when latency and errors rise with app runtime saturation while DB metrics remain healthy.
  • Fix or scale the DB tier first when DB query p95, held connections, or Postgres backends rise before app saturation.

8. Interview Questions

  • What are the four golden signals?
  • What is the difference between an SLI and an SLO?
  • When would you use traces instead of metrics?
  • How do you know a database is the bottleneck?
  • When can retries make incidents worse?
  • What makes a good alert?
  • How do you decide whether to scale horizontally?

9. Strong Interview Answer Pattern

  1. State the symptom clearly.
  2. Name the first dashboards or signals you would check.
  3. Describe how you would confirm the bottleneck.
  4. Give the immediate mitigation.
  5. Give the long-term fix.

10. Practice Routine

  1. Run a baseline scenario and explain each dashboard aloud.
  2. Run a slow DB or pool stress scenario.
  3. Predict the bottleneck before opening Jaeger.
  4. Use logs and traces to confirm your hypothesis.
  5. Choose whether to scale, tune, or mitigate.
  6. Summarize the incident in 2 to 3 minutes like an interview answer.
